How it works

Built differently,
by design.

Elimu is not a content library. It is a system that maps what each student knows, finds the gaps, and closes them — automatically. Here is exactly how.

01 Knowledge graph 02 RAG pipeline 03 Progress monitor 04 Data isolation
01
Knowledge graph

The DAG — a map of what each student knows

Every student has a personal Directed Acyclic Graph (DAG) — a living map of the CBC curriculum where each node is a concept and each edge is a prerequisite relationship. A concept only unlocks when every concept it depends on has been mastered. Not attempted. Mastered.

Grade 7 Mathematics — Integers strand

→

Prerequisites enforced

A student cannot reach Word Problems without first mastering Adding, Subtracting, and Mixed Operations. The graph enforces this automatically — no teacher configuration needed.

⚖

Weighted mastery scoring

Each concept ends with a 3-question quiz. Q1 tests recall (20%), Q2 tests application (30%), Q3 tests reasoning (50%). A student must score 80% or above to unlock the next concept. Getting easy questions right is not enough.

↗

Automatic unlocking

When a student passes a quiz, the system immediately checks every downstream concept. Any concept whose prerequisites are now all mastered switches from locked to available — without teacher input.

02
RAG pipeline

AI that explains the exact gap, not a generic recap

When a student fails a quiz, the system does not simply show them the same content again. It uses a Retrieval-Augmented Generation (RAG) pipeline to build a personalised re-explanation — anchored to the CBC syllabus, targeted at the specific questions they got wrong.

✗
Student fails quiz e.g. Gets Q2 and Q3 wrong on "Word Problems"
↓
⌕
Knowledge base retrieval System finds the most semantically relevant content chunks from the CBC curriculum using vector similarity search
↓
◈
Prompt is assembled The concept body, retrieved chunks, student's wrong answers, and attempt number are combined into a targeted prompt
↓
✦
Explanation is generated The AI produces a new explanation — using Kenyan everyday examples, CBC language, and the student's specific misconceptions as anchors
↓
↺
Student re-reads and retries On each subsequent failure the explanation goes deeper. By attempt 3 it starts from the foundational prerequisite, not the concept itself
📌

Always CBC-anchored

Generated content never introduces concepts outside the CBC syllabus for that grade level. The retrieval step ensures the explanation stays within the curriculum boundary.

⚡

Pre-generated in background

The moment a quiz result is returned, the system begins generating the remedial content in the background. By the time a student clicks "Try again" the explanation is ready — no waiting.

📈

Gets smarter over time

Every AI-generated explanation that led to a student mastering a concept is logged. This data is used to fine-tune the underlying model on explanations that actually work for Kenyan students.

03
Progress monitor

Teachers see who needs help before the end of term

The teacher dashboard is not a report. It is an intervention queue. The system runs continuously in the background — scanning every student's activity, detecting patterns that signal a student is in trouble, and surfacing specific recommended actions.

7 automatic flag types

Repeated failure, prerequisite gap, no activity, score regression, low engagement, unusually fast completion, and consistent reasoning failure — all detected automatically, no teacher setup.

🗺

Class heatmap

A colour-coded grid of every concept versus every student. Green means mastered, amber means in progress, red means stuck. A teacher can see the entire class's knowledge state in one view.

✏

Custom learning paths

Teachers can override the graph's default path for any student — for example, routing a student who missed two weeks back to an earlier concept — with a stated reason that is logged.

04
Data isolation

Every school's data is completely isolated

Elimu uses a schema-per-tenant architecture. When a school joins the platform, a dedicated PostgreSQL schema is created for them. Their students, teachers, progress data, and intervention records never share a database table with any other school.

Database architecture — schema-per-tenant

🏫

One subdomain per school

Each school accesses the platform at their own subdomain — e.g. nairobi-academy.elimu.co.ke. Requests are automatically routed to that school's schema. No configuration needed by the school.

🔒

Strict data boundaries

A teacher at one school cannot see — even accidentally — the students, scores, or flags from another school. The isolation is enforced at the database level, not just the application level.

📚

Shared curriculum, separate data

The CBC curriculum content — concepts, questions, strands — lives in a shared schema accessible to all schools. Only student activity data is isolated. This means curriculum updates apply everywhere instantly.
